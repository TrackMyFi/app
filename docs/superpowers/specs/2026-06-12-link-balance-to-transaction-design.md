# Link balance-history rows to their generating transaction

**Date:** 2026-06-12
**Status:** Approved

## Problem

On the Accounts page ([src/pages/Accounts.vue](../../../src/pages/Accounts.vue)), each account card shows a
Balance History table. Some balance snapshots are generated automatically when a transaction is
recorded, but there is currently no way to get from such a row back to the transaction that created
it. We want a balance row that originated from a transaction to expose an affordance that opens that
transaction's details, read-only, in a modal.

## Existing relationship

The link runs transaction → balance, not the reverse:

```sql
-- src-tauri/migrations/0002_transactions.sql
generated_balance_id     INTEGER REFERENCES account_balance(id) ON DELETE SET NULL,
generated_balance_to_id  INTEGER REFERENCES account_balance(id) ON DELETE SET NULL, -- transfer destination
```

A balance snapshot is generated by **at most one** transaction:

- A normal transaction sets `generated_balance_id` to the snapshot it created on its account.
- A transfer sets `generated_balance_id` (source account snapshot) **and** `generated_balance_to_id`
  (destination account snapshot). Both of those balance rows therefore link back to the *same*
  transaction.
- Manually-entered balance snapshots have no linking transaction (the common case).

The frontend `AccountBalance` type currently has no back-reference, and the transactions store is
paginated, so the link cannot be resolved reliably on the client today.

## Design

### 1. Data layer — resolve the link in the balance query

Add an optional back-reference to the balance model and populate it with a `LEFT JOIN`.

- **`src-tauri/src/models.rs`** — add `linked_transaction_id: Option<i32>` to `AccountBalance`.
  ts-rs regenerates `src/lib/types/AccountBalance.ts`.
- **`src-tauri/src/commands/accounts.rs`** — update the `SELECT` in both `list_all_balances` and
  `list_account_balances` (they share `row_to_balance`) to:

  ```sql
  SELECT b.id, b.account_id, b.balance, b.recorded_at, t.id
  FROM account_balance b
  LEFT JOIN txn t
    ON t.generated_balance_id = b.id OR t.generated_balance_to_id = b.id
  ORDER BY b.recorded_at
  ```

  `row_to_balance` reads the new column at index 4 into `linked_transaction_id`. Because the link is
  one-to-one, the join cannot multiply balance rows.

The Accounts page only consumes `list_all_balances` (via `listAllBalances` →
`store.allBalances`), but `list_account_balances` is updated too so the model stays consistent across
both code paths.

### 2. Fetch the transaction for the modal — lazy, by id

The all-balances payload ships only the linked transaction *id*, keeping it lean. The full
transaction is fetched on click.

- **`src-tauri/src/commands/transactions.rs`** — add `get_transaction(conn, id) -> Transaction` inner
  fn plus a `get_transaction_cmd` wrapper, registered in the Tauri handler.
- **`src/lib/api/transactions.ts`** — add `getTransaction(id) => invoke<Transaction>('get_transaction_cmd', { id })`.

### 3. UI

- **`src/components/BalanceRow.vue`** — when `balance.linkedTransactionId != null`, render a receipt
  icon button (`i-lucide-receipt`, `variant="ghost"`, `size="xs"`) in the actions cell, before the
  Edit and Delete buttons. Clicking it emits `view-transaction` with the transaction id. Rows with no
  linked transaction render nothing extra. The button is only shown in the non-editing (default) row
  state.
- **`src/pages/Accounts.vue`** — owns the modal and fetch state, mirroring how it already owns the
  account add/edit modal. Listens for `@view-transaction` from `BalanceRow`, fetches the transaction
  via `getTransaction(id)`, stores it in a ref, and opens a `UModal` whose body is the new
  `TransactionDetail` component.
- **`src/components/TransactionDetail.vue`** — new presentational, read-only component. Props: the
  `Transaction`. Renders: date, description, amount (colored by type — income green, expense red),
  type, category, account name (resolved via the accounts store), `→ transfer account` when
  `type === 'transfer'`, contribution flag, and import source. No edit controls.

Out of scope (deferred): an "Open in Transactions" button that navigates to the Transactions page.

## Testing

- **Rust (`src-tauri/src/commands/accounts.rs` tests):** extend the `list_all_balances` coverage to
  assert `linked_transaction_id` is `Some(txn_id)` for a transaction-generated balance and `None` for
  a manually-added balance. Assert both sides of a transfer resolve to the same transaction id.
- **Rust (`src-tauri/src/commands/transactions.rs` tests):** add a `get_transaction` test (round-trip
  a created transaction by id; error/empty behavior for a missing id consistent with existing
  commands).
